# Not All Data are Created Equal: Data and Pointer Prioritization for Scalable Protection Against Data-Oriented Attacks

# 摘要

​ 针对大多数操作系统中最先进的防御措施，面向数据的攻击变得越来越现实和有效。这些攻击操纵内存驻留数据对象（数据和指针），而不改变程序的控制流。由于对所有数据对象的过度检测，用于保护数据和指针的基于软件和硬件的对策遭受性能瓶颈。在这项工作中，我们提出了一个数据和指针优先级（DPP）框架，利用基于规则的启发式方法自动从应用程序中识别敏感内存对象，并利用现有对策仅保护那些敏感数据。我们使用 Linux Flaw Project 数据集、Juliet Test Suite 和五个真实程序（用于演示面向数据的攻击）来评估我们框架的正确性。我们的实验表明，DPP 可以通过优先考虑总数据对象的 3-4% 来识别我们测试的应用程序中易受攻击的数据对象。我们对 SPEC CPU2017 Integer 基准套件的评估表明，与默认 ASan 相比，DPPenabled AddressSanitizer (ASan) 可以将性能（在吞吐量方面）提高约 1.6 倍，并将运行时开销减少约 70%，同时保护所有优先数据对象。

# 介绍

  • 基于软件的对策：

    • 数据流完整性 (DFI)
    • 数据空间随机化 (DSR)
    • 内存标记技术
    • 缺陷：运行时开销大

  • 基于硬件的对策

    • HDFI
    • Intel 的控制流执行技术
    • ARM 指针身份验证
    • 缺陷：仅限于某些平台，开销不可忽略

  • 本文的方法：通过 DPP 框架自动识别敏感数据对象并确定其优先级

    • 挑战
      • 一套具有代表性、覆盖范围广泛的规则具有挑战性
      • 评估规则的准确性也具有挑战性
    • 贡献
      • 我们提出了一个适应性强且与平台或应用程序无关的 DPP 框架，用于自动识别敏感数据对象并对其进行优先级排序。我们定义了七种基于规则的启发式方法来进行识别和优先级排序。
      • 我们的评估表明，DPP 可以通过优先考虑总数据对象的 3-4% 来检测来自 Linux 缺陷项目 [19]、朱丽叶测试套件 [5,47] 和五个实际程序的易受攻击的数据对象
      • 我们通过 SPEC CPU2017 Integer 基准套件进行的性能评估表明，与 ASan（第 5.4 节）相比，DPP 在吞吐量方面将性能提高了约 1.6 倍，并将运行时间缩短了约 70%。我们的代码位于 https://github.com/salmanyam/dpp-llvm

# 背景

  • Data-Oritented Attacks

    • 可以在完全精确的 “静态” CFI 存在的情况下帮助控制数据攻击或实现图灵完备的表达能力
    • DOA 的关键是对两种数据的策略操作：i) 数据变量或对象，以及 ii) 数据指针
      • 覆盖安全关键数据变量或对象会导致程序行为改变或无意的数据泄漏
      • 覆盖数据指针即可实现 DOP，实现图灵完备的攻击

  • DOA 的防御手段

    • pointer-based，bound-checking ，object-based bound checking ，pointer integrity，DFI

      ，DSR，YARRA , PT-Rand ,Orpheus

    • 高运行时开销，大多数这些保护机制通常不适用于保护内存驻留数据对象

    • 因此，有必要对敏感数据对象进行识别并确定优先级，以改进这些保护机制并使其实用化

# Data and Pointer Prioritization

  • 敏感数据对象

    • 攻击者可以通过外部操作进行控制
    • 允许攻击者更改程序流程或选择执行路径

  • 两个主要操作

    1） 污点跟踪以识别可能由攻击者控制的数据对象

    • 外部源（例如网络、文件系统或用户输入）接收输入的系统或库 I/O 函数作为初始污点源

    2） 检测并确定对象的优先级

# 基于规则的优先级

​ DPP 使用基于规则的启发式方法自动检测可能导致漏洞的敏感数据对象，基于至少 9 个面向数据的漏洞利用 和 20 个 CVE 披露，将内存破坏的类型分为四类：i）控制更改 - 数据对象及其指针的损坏旨在改变程序行为，ii）基于邻近性 - 指针旨在损坏附近的缓冲区，iii）错误 - 其中数据指针的转换错误，违反了预期的指针语义，并导致内存损坏；或者数据对象有错误的边界条件，以及 iv) 无保护 —— 指针被分配为无边界，七种启发式规则如下：

# 数据流跟踪

目的：识别所有攻击者控制的数据（即来自外部输入的所有可访问数据）并污染它们。

方法：利用过程间 SVF 对 LLVM IR 进行分析，以构建静态值流图 (SVFG)

# 污点分析

目的：识别程序中的所有污点来源，并考虑简单和复杂的污点传播场景，在整个程序中传播污点

方法：1. 首先识别受污染的来源（一般来自网络、文件系统或用户输入），包括标准库函数，例如 read、recv、getc、recvmsg、scanf、fscanf、fread 和 fgets，以及标准库中的输入函数的常见包装函数。

​ 2. 通过遍历受污染节点的所有后继节点来通过 SVFG 传播污染

同时，需要解决 SVFG 的以下限制：

  1. 当传递给分配器的参数被污染时，SVFG 不会污染动态分配的内存。

  2. 当指针使用受污染的值或受污染输入的指针算术结果进行索引时，SVFG 不会污染底层对象。

  3. SVFG 不会跟踪函数（例如 memcpy）何时将一个参数的值存储到另一个参数

# 评估

  1. DPP 对安全关键数据对象进行优先级排序的能力如何？

    • 具有 DPP 的 ASan 可以像默认 ASan 一样检测两个数据集中的所有内存错误。然而，具有 DPP 的 ASan 可以通过仅保护优先数据对象来检测内存错误。此评估展示了 DPP 对安全关键数据对象进行优先级排序的能力

  2. DPP 对敏感数据对象进行排名的效率如何？个别规则对优先级排序的准确性有什么影响？

  3. 剩余数据对象有多敏感？

    • 通过案例分析 ghttpd-1.4 和 nullhttpd-0.5.0，DPP 未优先考虑的数据对象确实不敏感

  4. DPP 可以实现多少性能提升？DPP 的最终结果如何适合实施实际保护？

    • 启用 DPP 的 ASan 相比默认的 ASan，性能提高了 1.6 倍，开销减少了 30%

    • 启用 DPP 的 ASan 相比默认的 ASan，运行时间开销减少了约 70%

# 总结

​ 在本文中，作者提出了一种自动优先级框架，用于识别和保护敏感的内存驻留数据对象，以防止面向数据的攻击。总体结果表明，简单的基于规则的启发式方法是有效的。我们的漏洞利用和漏洞驱动的基于规则的启发式方法可以在必要时灵活地添加新规则。使用 Linux Flaw Project [19]、Juliet Test Suite [5, 47] 和实际程序进行的实验评估表明，可以成功识别易受攻击的数据对象。与 ASan 相比，DPP 将性能提高了 ∼1.6 倍，并将运行时开销降低了 70%。作者认为并非应用程序中的所有数据对象都需要保护，我们可以对它们进行优先级排序。这种提出的优先级方案是新的，使我们的方法不同于传统的保护范例。
